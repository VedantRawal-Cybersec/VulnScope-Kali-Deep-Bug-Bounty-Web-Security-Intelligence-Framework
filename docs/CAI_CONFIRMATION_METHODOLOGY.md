# CAI Zero-Impact Confirmation Methodology

This document defines the VulnScope CAI confirmation layer. It is intentionally scoped to evidence verification and confidence scoring. It does not add exploit payloads, authentication bypass logic, production data modification, destructive requests, or high-volume active testing.

## Scope

Allowed:

- Passive asset and endpoint review.
- Already-collected HTTP request/response evidence analysis.
- Safe structural comparison evidence generated by approved non-destructive probes.
- Confirmation scoring, deduplication, suppression, and reporting.

Not allowed:

- Modifying production data.
- Sending destructive POST, PUT, PATCH, or DELETE payloads.
- Exploit chains, brute force, denial-of-service, or authentication bypass behavior.
- Calling any candidate a confirmed vulnerability without a captured comparable evidence artifact.

## Source Fit Evaluation

| Source | Fit | Implementation decision |
|---|---|---|
| OWASP API Security Top 10 2023, API1 BOLA | High | Keep IDOR/BOLA eligibility restricted to endpoints that receive object identifiers and act on account/object resources. Static help and marketing paths are suppressed. |
| OWASP Web Security Testing Guide | High | Use WSTG-style evidence-driven reporting and stable scenario identifiers as inspiration for traceable evidence artifacts. |
| OWASP ASVS | High | Require authentication/session artifacts before Auth/JWT issues are eligible. URL words alone are not evidence. |
| MDN CORS documentation / Fetch CORS model | High | Require captured `Access-Control-Allow-Origin` and `Access-Control-Allow-Credentials` values before surfacing CORS leads. |
| CAI public framework concept | Medium | Adopt modular agent architecture and autonomous evidence reasoning, but do not import offensive behavior or autonomous exploit actions. |

## Stage 0: Deduplication

Raw candidates are grouped by:

```text
(vulnerability_type, url_path_without_query_string)
```

Only one representative is kept per group unless distinct parameters clearly represent different backend code paths. The report always includes:

```json
{
  "raw_count": 0,
  "unique_count": 0,
  "dedup_ratio": "raw -> unique",
  "duplicates_suppressed": 0
}
```

## Stage 1: Eligibility Gate

A candidate must pass class-specific eligibility before scoring.

- IDOR/BOLA requires object/account identifier evidence on a non-static endpoint.
- SQLi requires observable data lookup variation or server-side error evidence.
- CSRF requires state-changing endpoint evidence.
- SSRF/Open Redirect requires redirect/fetch behavior evidence.
- CORS requires API/data endpoint context and captured CORS headers.
- Auth/JWT requires actual token, cookie, session, or Authorization evidence.
- Safe Parameter Review requires safe marker reflection or structural behavior evidence.

Failures are logged to `suppressed_noise` with a one-line reason.

## Stage 2: Evidence Requirements

A surfaced candidate must include one captured comparable artifact:

- structural response diff,
- cross-session or cross-account comparison,
- server-side error signature,
- redirect/fetch target confirmation,
- header inspection.

Keyword matches, path names, HTTP 200 responses without baselines, and forms without behavior evidence are not sufficient.

## Stage 3: Confidence Scoring

The scoring formula is:

```text
confidence_score = evidence_strength*0.4 + reproducibility*0.3 + impact_estimate*0.3
```

Confidence tiers:

- LOW: goes to `needs_more_signal`; not shown as a dashboard finding.
- MEDIUM: shown as `REVIEW LEAD`.
- HIGH: eligible for `CONFIRMED` only when direct comparable evidence exists and a baseline/reproduction condition is met.

## Stage 4: Dashboard Output

Every surfaced item includes:

```json
{
  "vulnerability_type": "...",
  "url_path": "...",
  "dedup_group_size": 1,
  "eligibility_check": "passed: ...",
  "evidence_type": "structural response diff | cross-session/cross-account comparison | server-side error signature | redirect/fetch target confirmation | header inspection",
  "evidence_detail": "actual captured artifact",
  "confidence": "medium | high",
  "confidence_score": 0.0,
  "score_breakdown": {
    "evidence_strength": 0.0,
    "reproducibility": 0.0,
    "impact_estimate": 0.0,
    "formula": "evidence_strength*0.4 + reproducibility*0.3 + impact_estimate*0.3"
  },
  "control_comparison_result": "baseline/control result",
  "false_positive_risk_notes": "..."
}
```

## Modular Agent Mapping

| Agent | Current role in VulnScope |
|---|---|
| Reconnaissance Agent | Top102 safe runners and passive inventory. |
| Parameter Analysis Agent | Adaptive safe parameter family review. |
| Hypothesis Generation Agent | Existing safe parameter and evidence card candidates. |
| Non-Intrusive Verification Agent | Existing approved safe/canary comparisons only. |
| Confidence & Evidence Engine | `finding_confirmation_engine.py` and `cai_confidence_policy.py`. |
| Deduplication & Prioritization Agent | Confirmation engine deduplication and confidence gate. |
| Report Generation Agent | Final confirmation dashboard and data bundle. |

## Safety Invariants

1. Default to skepticism.
2. No evidence artifact means no surfaced finding.
3. LOW confidence never reaches the final dashboard.
4. MEDIUM is a review lead, not a vulnerability.
5. CONFIRMED requires direct comparable evidence and high confidence.
6. Suppressed items are logged for tuning, not reporting.
